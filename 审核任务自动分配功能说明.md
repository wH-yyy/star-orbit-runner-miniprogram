# 审核任务自动分配功能说明

## 功能概述

系统已实现自动分配审核任务功能。当用户提交跑步记录后，系统会自动将该记录分配给一个工作人员进行审核。每个工作人员登录后只能看到分配给自己的审核任务（包括待审核、通过和未通过的记录）。

## 核心功能

### 1. 自动分配机制

- **触发时机**: 用户通过小程序提交跑步记录时自动触发
- **分配算法**: 使用负载均衡算法（Round Robin），选择当前待审核任务最少的工作人员
- **分配对象**: 只分配给状态为 `active`（激活）的工作人员

### 2. 权限隔离

- 工作人员登录后只能查看分配给自己的记录
- 无法查看或审核其他工作人员的任务
- 基于工作人员的唯一标识 `_id` 进行权限控制

### 3. 工作负载平衡

- 系统统计每个工作人员当前的待审核任务数（status=0）
- 自动选择任务最少的工作人员进行分配
- 确保审核任务在所有工作人员之间均衡分布

## 数据库设计

### RunningRecords 集合新增字段

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `assignedStaffId` | String | 分配的工作人员ID（staff表的_id） |
| `assignTime` | Date | 任务分配时间 |

### staff 集合关键字段

| 字段名 | 类型 | 说明 |
|--------|------|------|
| `_id` | String | 工作人员唯一标识 |
| `status` | String | 工作人员状态（active/inactive） |
| `username` | String | 用户名 |

## 技术实现

### 后端实现

#### 1. staff-api 云函数新增接口

**接口**: `audit/assignTask`

**功能**: 为指定记录分配审核工作人员

**参数**:
```javascript
{
  action: 'audit/assignTask',
  recordId: 'record_id' // 跑步记录ID
}
```

**返回**:
```javascript
{
  code: 200,
  message: 'Task assigned successfully',
  data: {
    recordId: 'record_id',
    assignedStaffId: 'staff_id', // 分配的工作人员ID
    staffWorkload: 5 // 该工作人员当前待审核任务数
  }
}
```

**分配逻辑**:
1. 检查记录是否存在
2. 如果已分配，直接返回当前分配结果
3. 查询所有激活状态（status='active'）的工作人员
4. 统计每个工作人员的待审核任务数
5. 选择任务最少的工作人员
6. 更新记录的 `assignedStaffId` 和 `assignTime` 字段

#### 2. staff-api 云函数修改接口

**接口**: `audit/getRecords`

**新增参数**: `staffId`

**功能**: 如果传入 `staffId`，则只返回分配给该工作人员的记录

**查询条件**:
```javascript
{
  assignedStaffId: staffId, // 只查询分配给指定工作人员的记录
  status: ..., // 其他筛选条件
  ...
}
```

#### 3. uploadRunningRecord 云函数修改

在用户提交跑步记录成功后，自动调用任务分配接口：

```javascript
// 创建记录
const dbResult = await db.collection('RunningRecords').add({ data: recordData })

// 自动分配审核任务
const assignmentResult = await cloud.callFunction({
  name: 'staff-api',
  data: {
    action: 'audit/assignTask',
    recordId: dbResult._id
  }
})
```

## 使用流程

### 1. 用户提交记录

```
用户 → 小程序 → uploadRunningRecord云函数
  ↓
创建RunningRecords记录
  ↓
调用 staff-api (audit/assignTask)
  ↓
自动分配给工作人员
```

### 2. 工作人员审核

```
工作人员登录 → 审核页面
  ↓
调用 getAuditRecords (staffId=当前工作人员ID)
  ↓
只显示分配给自己的记录
  ↓
进行审核操作
```

## 注意事项

### 1. 工作人员状态管理

- 只有 `status='active'` 的工作人员才会被分配任务
- 如果需要暂停某个工作人员接收新任务，将其状态改为 `inactive`
- 修改状态不影响已分配的任务

### 2. 错误处理

- 如果没有可用的工作人员（所有工作人员都是 inactive），分配会失败
- 分配失败不会影响跑步记录的创建，记录会保存但 `assignedStaffId` 为空
- 可以后续手动分配或通过批量分配功能补救

### 3. 性能考虑

- 分配算法会查询所有激活工作人员的待审核任务数
- 如果工作人员数量较多，可能影响性能
- 建议将工作人员数量控制在合理范围（如 10-50 人）

### 4. 权限安全

- 前端传入的 `staffId` 应该与登录状态绑定
- 后端应该验证请求者的身份（当前实现信任前端传入的 staffId）
- 生产环境建议在云函数中通过 context 获取登录态验证身份

## 未来改进方向

1. **权限增强**: 在后端通过登录态验证工作人员身份，避免前端伪造 staffId
2. **分配策略优化**: 支持更多分配策略（如按地区、按专业领域等）
3. **重新分配**: 支持管理员手动调整任务分配
4. **统计报表**: 展示每个工作人员的审核量、通过率等统计数据
5. **负载预警**: 当某个工作人员任务过多时自动预警
6. **批量分配界面**: 为管理员提供批量分配未分配记录的界面

## 测试建议

### 1. 自动分配测试

- 创建多个 active 状态的工作人员
- 提交多条跑步记录
- 验证记录是否均衡分配
- 检查 `assignedStaffId` 和 `assignTime` 字段是否正确

### 2. 权限隔离测试

- 使用不同工作人员账号登录
- 验证每个工作人员只能看到自己的任务
- 尝试修改 URL 参数或请求参数，验证无法访问他人任务

### 3. 边界情况测试

- 没有 active 工作人员时提交记录
- 所有工作人员都有大量待审核任务时的分配策略
- 记录已分配后重复调用分配接口的处理
